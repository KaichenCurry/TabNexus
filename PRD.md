# TabNexus 产品需求文档（PRD）

> **一句话定位：让用户、浏览器和 AI Agent 共享同一份任务上下文的 Tab 工作台。**
> 卡片给人看，结构给 Agent 用——核心价值是共享任务上下文本身，AI 是把整理成本压到零的加速器，不是价值前提。

| 项目 | 内容 |
|---|---|
| 产品名 | **TabNexus**（曾用名 TabFlow，已统一） |
| 版本 | v0.4 |
| 作者 | Chen |

## 1. 背景与问题

### 1.1 AI 时代的标签爆炸：三个新增量

1. **Agent 替人开页**：Codex、Cursor、Claude Code 等工具的 Agent 主动打开或要求跳转浏览器，产生大量"不是自己开的、但不敢关"的标签。
2. **AI 产出以 HTML 呈现**：AI 生成的 PRD、报告、原型以 HTML 交付，每份产出都是一个新标签。
3. **人机来回跳转**：用户在 AI 工具与浏览器之间高频切换，任务上下文被切碎。

结果不只是视觉混乱，而是**认知负担**：记不清"这个标签是为哪件事开的"，任务思路随标签堆积而丢失。

### 1.2 上一代方案的坟场问题

Toby、Workona、OneTab 验证了收纳需求，但共同宿命是**保存即遗忘**。根因：只解决"标签放哪"，没解决"任务怎么推进"。本产品的立足点：**标签背后有任务意图，用户回来是为了继续任务，Agent 进来是为了接住上下文。**

---

## 2. 目标用户与锚定场景

**目标用户**：深度使用 AI 工具的知识工作者（PM、研究员、分析师、独立开发者）。以任务/Case 为单位工作，单任务 10–30 个标签，AI 工具与浏览器双开。

**锚定场景：调研一家公司。** 两小时开十几个页面（市场/产品/技术/财务/观点各有角色），在浏览器里却是无差别平铺。TabNexus 把它们收进一个 Workspace，AI 建议分组、用户确认，得到结构清晰的研究看板；关闭所有标签也不丢上下文，可整体恢复；**结构化导出直接喂给 Codex/Cursor 继续干活**。

---

## 3. 产品架构：三个核心层

三层共用同一个 Workspace 数据模型，是同一份任务上下文的三种访问方式。

| 层 | 名称 | 解决什么 | 落地节奏 |
|---|---|---|---|
| L1 | 标签与 Workspace 管理 | 信息进得来 | M1 |
| L2 | 任务思路管理 | 思路理得清 | 卡片=M1；流程图/关系/进度=M2 |
| L3 | Agent 协作接口 | 上下文流得动 | **导出闭环=M1**；完整 MCP=M3 |

**关键判断**（开发中不可动摇）：
1. 核心价值是共享任务上下文，不押注 AI 聚类准确率（分流机制见 §5）。
2. 手动拖拽是纠正手段，不是主路径——不能让用户回到全手动整理。
3. L1/L2 会被平台逐步收编，L3 是长期护城河。
4. MVP 主线只有一条（见 §6），其余一律进 M2/M3，开发中不得加需求。

---

## 4. 数据模型（P1 修改：统一内容对象）

卡片不只是网页。Workspace 承载任务的全部资料形态：

```
workspace = {
  id, name,
  groups: [{ id, name, color }],
  cards:  [{
    id,
    type,          // 'web' 网页 | 'note' 笔记 | 'html' 本地HTML文档 | 'report' 报告 | 'agent' Agent产出
    title, url,    // note 类型可无 url
    favicon, note,
    status,        // 待读 | 已读 | 已采用（轻进度，M2 启用）
    groupId,
    source,        // 'user' | 'ai' | 'agent'（谁添加的）
    flow: { x, y } // M2 流程图坐标
  }],
  edges: [{ fromCardId, toCardId, label? }]   // M2 启用
}
```

- 任一视图/接口的修改作用于同一对象；分组颜色贯穿卡片与（M2 的）流程图节点
- `type` 与 `source` 从 M1 起就写入（导出给 Agent 时需要），UI 差异化展示可后置

---

## 5. AI 验证门槛（P0 修改：接受率分流）

M0 用 3~5 组真实标签样本实测 DeepSeek 聚类，以**用户接受率**（未被修改的分组归属占比）分流：

| 接受率 | 结论 | M1 交互形态 |
|---|---|---|
| ≥ 80% | 聚类可靠 | **自动分组**为默认（可撤销） |
| 60% – 79% | 可用但不可靠 | **只提供建议**：AI 给分组草稿，用户确认后生效 |
| < 60% | 不可用 | **重新设计**：迭代 prompt / 换模型 / 改为模板+手动，再测 |

无论落在哪一档，M1 都照常开工——变的是 AI 的交互形态，不是产品是否成立。无 API key 或调用失败时，始终降级为域名分组。

## 6. MVP 标准流程（M1 唯一主线）

```
采集当前标签
    ↓
进入收件箱
    ↓
DeepSeek 推荐分组
    ↓
用户确认或调整
    ↓
形成 Workspace
    ↓
保存、关闭、恢复
    ↓
导出结构化上下文给 Agent
```

关键路径：从"标签混沌"到"结构出现"≤ 30 秒、两次点击；从 Workspace 到"Agent 可用的上下文"一次点击。

## 7. 版本范围

### M0 · 验证两个核心假设（当前阶段）

| # | 假设 | 验证方法 | 通过标准 |
|---|---|---|---|
| H1 | DeepSeek 能准确整理真实标签 | 聚类脚本 + 3~5 组真实标签样本，人工判定接受率 | 按 §5 分流表决定 M1 形态 |
| H2 | 结构化 Workspace 能提高 Agent 工作效率 | 同一批资料以"平铺 URL 列表" vs "结构化 Markdown（分组+备注+类型）"分别喂给 Codex/Cursor 完成同一任务，对比输出质量与来回轮次 | 结构化版本明显更优（主观判定+轮次减少），否则重新设计导出格式 |

### M1 · 最小可用产品

- 采集当前窗口标签（Open Tabs 侧栏拖入 + 一键全部收集，可选收集后关闭原标签）
- AI 推荐分组（按 §5 分流的形态；结果可整体撤销）
- 用户确认和拖拽调整（分组新建/重命名/换色/删除；卡片备注/删除/打开）
- 卡片 Workspace（Toby 式 collections；多 Workspace 隔离与切换）
- 保存、关闭与恢复（storage.local 持久化；按分组/整个 Workspace 恢复；重启不丢）
- **导出给 Agent**：一键把 Workspace 导出为 Markdown（人可读、可直接粘给 Codex/Cursor/Claude Code）与 JSON（结构完整含 type/source），复制到剪贴板或下载
- 设置页（DeepSeek key 校验、隐私说明：仅发送标题+URL）

**M1 验收清单**

- [ ] Open Tabs 侧栏实时显示当前窗口标签；拖入分组/全部收集可用；20 个标签 < 2s；重复 URL 去重
- [ ] AI 推荐分组按 M0 分流形态工作；过程态清晰；一键撤销；无 key 降级域名分组仍可用
- [ ] 卡片拖拽换组有落点反馈；分组管理完整；删除分组时卡片回收件箱
- [ ] 第二个 Workspace 数据隔离；切换 < 500ms
- [ ] 恢复 15 个标签不丢单；重启浏览器数据完好
- [ ] 导出 Markdown 粘进 Cursor/Codex 可直接作为任务上下文使用；JSON 可被脚本解析
- [ ] 界面视觉与 Claude Design 设计图一致

### M2 · 思路管理

- 流程图模式（节点/拖动/手动连线）
- 网页关系（连线标签、AI 建议连线）
- AI 推荐任务结构
- 研究进度管理（卡片三态：待读/已读/已采用）
- 最近关闭缓冲、搜索、手动补录

### M3 · Agent 双向协作

- MCP 接口（Native Messaging / 本地桥接）
- Agent 读取 Workspace（read_workspace）
- Agent 添加资料（add_card）
- Agent 写回报告（write_report）与建议任务结构（propose_structure）
- 人与 Agent 双向协作的完整闭环

### 暂时删除或降级（第一版不做）

完整流程图 · 自动生成复杂关系 · 完整 MCP 双向协作 · 云同步和账号 · 团队协作 · 网页全文读取 · 过于复杂的进度系统 · 新标签页劫持 · 上架材料

## 8. 成功指标（M1 dogfood 两周，手动记录）

北极星：**Workspace 7 日回访率**（度量坟场问题）。

| 指标 | 目标 | 不达标处置 |
|---|---|---|
| AI 分组接受率 | ≥ 70%（自动式）/ ≥ 80%（建议式的采纳率） | 迭代 prompt（M0 脚本回归）；仍不达标降档交互形态，不影响主闭环 |
| Workspace 7 日回访率 | ≥ 40% | 回到坟场问题重审 L2 假设，暂停 M2 |
| 恢复行为周频次 | ≥ 3 次 | — |
| 导出给 Agent 周频次 | ≥ 2 次 | H2 重新验证导出格式 |

## 9. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 聚类不准 | 整理成本上升（非致命：核心价值在共享上下文） | §5 分流；接受率驱动 prompt 迭代；域名分组兜底；可撤销 |
| 导出格式 Agent 用不好 | L3 价值验证失败 | H2 前置验证；格式按 Agent 实测反馈迭代 |
| 平台挤压（Chrome/AI 浏览器原生整理） | L1 价值稀释 | 重心压在 L2/L3 |
| MCP 桥接复杂度 | M3 延期 | M1 用导出闭环过渡，桥接后置 |
| 隐私顾虑（URL 外发） | 信任受损 | 默认降级可用；AI 显式开启；设置页明示数据边界 |
| DeepSeek 超时/变更 | 链路不稳 | 8s 超时+一次重试+降级；provider 可替换 |

## 附：设计基准

- UI 形态基准：**Toby 的 Workspace 形态**——左侧空间轨 + 白色侧栏、主区可折叠 collections + 双层卡片（favicon+标题/分割线/灰色小字）、右侧常驻 OPEN TABS 面板、珊瑚红点缀
- 意图色仅用于分组圆点与（M2）流程图节点
- 交互参考原型：同目录 `mockup.html`
- **最终视觉与页面规格以 Claude Design 产出的设计图为准**；与本 PRD 冲突时，交互逻辑以 PRD 为准、视觉以设计图为准
- 采集/存储底层参考 [Tab Out](https://github.com/zarazhangrui/tab-out)：MV3、chrome.tabs、全本地
